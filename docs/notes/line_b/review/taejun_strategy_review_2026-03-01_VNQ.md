### M5 — 비중조절 메커니즘 (MT_VNQ4 전면 정정)

#### 용어 정의 *(MT_VNQ4 확정)*

| 용어 | 정의 |
|------|------|
| **n** | 같은 시점에 매수 허가(PASS)된 신호(종목) 개수. 전략 여러 개가 동시에 PASS면 합쳐서 n으로 센다 |
| **비중 단위** | **전체 자산(현금+주식 포함) 대비 %** 로 고정 |
| **매수 우선순위** | **M_PASS 개수 > Y_PASS 개수 > target_net 높은 순** (점수는 수익률에 합산 금지) *(MT_VNQ5 §9-2 확정)* |

#### MMM(필수) 조건 *(MT_VNQ4 신설)*

매수 프로토콜은 **MMM 3개**가 모두 PASS해야만 진입 가능 (MMM 하나라도 FAIL이면 그 종목은 n 계산에서 제외).

> ⚠️ MMM = 거래대금 배수 조건 + 금 조건 + Polymarket 조건. M4000 프로토콜 §1~§3 참조.

#### M5-매수: T1~T4 배분 규칙 *(MT_VNQ4 정정: 기존 55/40/33 순차 → 99/50/33 총비중)*

| n (PASS 종목 수) | T1 | T2 | T3 | T4 | 비고 |
|-----------------|------|------|------|------|------|
| n=1 | **99%** | — | — | — | 나머지 1%는 현금 유지 |
| n=2 | **50%** | **50%** | — | — | |
| n=3 | **33%** | **33%** | **33%** | — | |
| n≥4 | **33%** | **33%** | **30%** | **나머지 전액(=4%)** | |
| T5~ (5개 이상) | 4개까지만 실제 매수(T1~T4), 5번째 이상은 **예약/대기 표시만**, 신규 매수 없음 | | | | |

> **예약(T5~) 10초 룰 *(MT_VNQ3 유지)*:** T5~ 예약은 **10초 경과 후** 조건 미부합이면 예약 취소.
> ⚠️ **"예약 10초"와 "주문 잔량 Fill Window 10초"는 서로 다른 개념**: 예약 = 주문 없음 / Fill Window = 주문이 이미 나간 상태.

#### 현금 부족 처리 *(MT_VNQ4 확정)*

- 매수 금액이 부족하면 **"살 수 있는 수량만"** 매수한다.
- T2/T3/T4로 더 많은 조건이 와도 **현금이 0이면 즉시 중단**한다.
- "돈이 없는데 다음 티어로 계속 계산/매수 시도"하는 오류를 원천 차단.

> ✅ **P-3 해결 (MT_VNQ6):** 여러 종목 동시 PASS 시 `reserved_cash`는 **각 주문 생성 시점에 개별 즉시 차감** (세트 완료 후 합산 아님). 주문 A 생성 시 A의 reserved_cash 즉시 차감 → 주문 B 생성 시 B의 reserved_cash 즉시 차감. CI-0-8 현금 예약 규칙과 일치.

#### Y/M/N 점수 규칙 *(MT_VNQ4 확정)*

| 등급 | 점수 | 의미 |
|------|------|------|
| **Y** | +0.5 | 최고 허가 |
| **M** | +0.3 | 중간 허가 (필수 조건 PASS) |
| **N** | 0 | 불허가 / 불충족 |

> **점수 용도 (2가지로 제한):**
> 1. **매수 우선순위 정렬(랭킹):** M PASS 개수 → Y PASS 개수 → 최종수익률 높은 순
> 2. **최종수익률에 절대 더하지 않는다.** 환산 규칙(예: 1점=0.1%)이 명시적으로 정의되지 않는 한 점수를 수익률에 합산 금지 — M80/M5에서 "단위 혼용 버그" 방지.

#### 최종수익률 하한 *(MT_VNQ4 확정)*

계산된 최종수익률이 **0.8% 미만이면 매수를 중단**한다.
- 최종수익률 ≥ 0.8% → 진입 허가
- 최종수익률 < 0.8% → 매수 불허(FAIL)

**현재 구조의 문제:**
- 각 전략 `generate_signal`은 독립적으로 size를 결정 (고정값)
- 전략 간 당일 조건 부합 카운팅 공유 메커니즘 없음

**해결 방향:**
```python
class Orchestrator:
    def __init__(self, strategies, portfolio):
        self._daily_match_count: dict[date, int] = {}  # 날짜별 조건 부합 누적

    def get_entry_size(self, today: date, signal_rank: int) -> float:
        # M5 T1~T4 순차 계산 방식 (개정)
        # T1=55%, T2=나머지(45%)x40%, T3=나머지x33%, T4=나머지 전액
        remaining = 1.0
        rates = [0.55, 0.40, 0.33, 1.0]
        for i in range(min(signal_rank, 4)):
            size = remaining * rates[i]
            remaining -= size
        return size  # signal_rank번째 진입분 비중

    def on_trade_executed(self, today: date):
        # M5: 완료 후 1개 차감
        self._daily_match_count[today] = max(
            self._daily_match_count.get(today, 0) - 1, 0
        )
```

- pending/partial 상태 주문이 있는 ticker+side에 대해 신규 주문을 금지한다(중복 주문 방지).

##### 12) 재시작/장애 복구 시 동기화 (유령 주문/중복 주문 방지)
- 엔진 시작 시 반드시:
  1. 브로커 open orders 조회
  2. 브로커 positions 조회
  3. 내부 OrderBook/Portfolio를 브로커 상태로 재구성
- 동기화 실패 시: 신규 매수 금지 + 알림(시스템 점검 모드)

##### 13) 수수료 모델 정정 (분할매도 x N 오해 방지)
- 기본: 수수료는 "거래대금 비율" 모델로 계산한다.
  - roundtrip_fee_pct = buy_fee_pct + sell_fee_pct
  - 분할매도(sell_splits)는 % 기준 총수수료를 증가시키지 않는다.
- 단, '주문당 최소수수료'가 존재하면:
  - min_fee 적용 여부를 옵션으로 두고,
  - 분할매도는 "절대비용"이 증가할 수 있으므로 별도 계산한다(옵션 ON일 때만).

##### 14) 데이터 결측/0값 금지 (0으로 체결되는 치명적 버그 방지)
- prices.get(ticker, 0) 사용 금지.
- 가격/지표 결측 시:
  - 해당 주문/신호는 스킵 + 알림
  - 전략 전체가 아니라 "해당 ticker만" 쿨다운(예: 5분)
